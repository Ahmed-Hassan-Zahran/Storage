  C# solution (clean / onion architecture, .NET 8):
  - Yoxel.Storage.Core — domain models (FileMetadata, StoredFile), abstractions (IFileStorage, IFileMetadataRepository, IStorageService, IStorageEventPublisher), orchestration (StorageService).   
  - Yoxel.Storage.Infrastructure — LocalFileStorage and S3-compatible S3FileStorage, EF Core + Npgsql StorageDbContext / repository, no-op event publisher, composition root.
  - Yoxel.Storage.Api — ASP.NET Core 8 Web API: FilesController, ApiKeyAuthHandler, problem+json ExceptionMiddleware, Swagger, health checks, 5 GB upload cap.
  - Dockerfile, docker-compose.yml (Postgres + MinIO + API), .dockerignore, .gitignore, Yoxel.Storage.sln.

  Key design decisions baked into code:
  - Storage abstracted behind IFileStorage; Storage:Backend config switches between Local and S3 without code changes.
  - Per-microservice X-Api-Key auth resolves to a tenant claim; the controller never trusts client-supplied tenant values.
  - Tenant isolation enforced in StorageService.GetOwnedAsync with a deliberately uniform 404 response so tenant existence doesn't leak.
  - Upload buffers to a DeleteOnClose temp file so SHA-256 + size are computed in a single pass without holding bytes in memory.
  - Soft delete via DeletedAt; queries filter out deleted rows.
  - Async lifecycle events stubbed via NoopStorageEventPublisher (logs only); replace with Kafka/RabbitMQ in production.

  To run:
  - Local backend, no deps: dotnet run --project src/Yoxel.Storage.Api → Swagger at http://localhost:5080/swagger. Default keys: dev-key-service-a, dev-key-service-b.
  - Full S3-compatible stack: docker compose up brings up Postgres + MinIO + API (auto-creates the yoxel-storage bucket).
